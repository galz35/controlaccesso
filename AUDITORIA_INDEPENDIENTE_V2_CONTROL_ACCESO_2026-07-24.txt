5. HALLAZGOS CRÍTICOS
======================================================================

C-01. DEV-LOGIN CONTINÚA ACTIVO EN EL DESPLIEGUE REAL

Evidencia:

- AuthController solo bloquea dev-login si NODE_ENV === “production”.
- nest/.env no contiene NODE_ENV.
- PM2 reporta node env: N/A.
- LoginPage sigue ofreciendo “Empleado” por carnet.
- AuthContext sigue llamando POST /auth/dev-login.
- sp_Login_Empleado crea usuario local si el carnet no existe.

Conclusión:
La corrección fue escrita, pero la condición de producción no se cumple en el
servidor activo. Por tanto, dev-login continúa habilitado.

Riesgo:
Cualquier persona que conozca o pruebe un carnet puede obtener un JWT. Además,
el SP puede crear usuarios locales para valores no verificados.

Corrección:

1. Definir NODE_ENV=production en PM2.
2. Reiniciar el proceso con entorno actualizado.
3. Eliminar la opción de carnet simple del frontend productivo.
4. Bloquear dev-login por configuración explícita, no solo NODE_ENV.
5. Usar una variable ALLOW_DEV_LOGIN=false por defecto.
6. Validar carnet contra la fuente corporativa antes de crear usuario local.
7. Añadir prueba que confirme HTTP 403 en producción.

C-02. SALIDA INDEPENDIENTE CORROMPE EL CONCEPTO DE OCUPACIÓN

Evidencia de código:

- registrarSalidaIndependiente inserta en tblRegistroAcceso.
- Usa TipoPersona=SALIDA_INDEPENDIENTE.
- La tabla asigna FechaEntrada=GETDATE().
- No asigna FechaSalida.

Evidencia de base:

- Existe 1 SALIDA_INDEPENDIENTE.
- Ese registro tiene FechaSalida nula.
- Por tanto aparece como pendiente.

Impacto:

- Una salida incrementa la cantidad de personas “dentro”.
- El reporte muestra el movimiento como entrada.
- El panel de pendientes puede mostrar una salida como persona pendiente.
- Los KPI quedan conceptualmente incorrectos.
- No se puede distinguir ocupación física real.

Corrección recomendada:

Opción preferida:
Crear tblMovimientosAcceso con TipoMovimiento ENTRADA/SALIDA/NOVEDAD y enlazar
eventos cuando exista correspondencia.

Opción mínima:
Agregar columnas TipoMovimiento y EsSalidaSinEntrada, y registrar FechaSalida
sin crear una FechaEntrada falsa.

Nunca debe quedar una salida independiente como FechaSalida=NULL.

C-03. CREDENCIALES EXPUESTAS Y SECRETOS CON FALLBACK

Evidencia:

- test/README.md conserva una contraseña de base escrita directamente.
- Esa información está en historial Git ya publicado.
- JwtModule y JwtStrategy todavía aceptan un secreto predeterminado conocido.
- main.ts solo muestra una advertencia; no detiene el servicio.

Aunque nest/.env tiene JWT_SECRET configurado actualmente, el código permite
iniciar inseguro en otro despliegue.

Corrección:

1. Rotar la contraseña expuesta.
2. Eliminarla de archivos actuales.
3. Evaluar limpieza del historial.
4. Quitar completamente el fallback JWT.
5. Lanzar error fatal si JWT_SECRET o SSO_SECRET faltan.
6. Agregar detección de secretos en CI.

A-03. NO EXISTE INTERFAZ PARA SALIDA INDEPENDIENTE

El endpoint está desplegado, pero no hay botón, formulario ni flujo visible.

Debe incluir:
- edificio
- búsqueda por carnet/código
- nombre confirmado
- observación obligatoria
- advertencia de que no se encontró entrada
- confirmación final

Antes debe corregirse el modelo descrito en C-02.

A-04. INSTALACIÓN 001-005 SIGUE SIN SER REPRODUCIBLE

Problemas:

- 004_create_procedures.sql define sp_Acceso_RegistrarEntrada sin
  @MotivoAcceso y @MotivoDetalle.
- 005_migration_v2.sql agrega columnas, pero no actualiza ese SP.
- Nest siempre envía ambos parámetros.
- Una instalación nueva terminará con contrato Nest/SQL incompatible.
- tblPersonalExterno y sus SP no están completos en los scripts base/migración.

Resultado:
Recuperar o instalar el sistema desde Git puede producir un backend que compila
pero falla al registrar entradas.

A-05. MIGRACIÓN V2 NO AGREGA FK A UNA COLUMNA YA EXISTENTE

La migración crea la FK únicamente dentro del bloque que agrega
EdificioIdDefecto.

En la base activa la columna ya existía, por lo que el bloque no se ejecutó.

Resultado comprobado:
Cantidad de FK sobre EdificioIdDefecto = 0.

Debe comprobarse la restricción por nombre/columnas independientemente de la
existencia de la columna.

A-06. FOTOGRAFÍAS TODAVÍA NO TIENEN PUBLICACIÓN SEGURA COMPLETA

AccesoService guarda archivos y devuelve /control-acceso-uploads/...
Nginx reenvía esa ruta al backend.
Nest no configura servicio de archivos estáticos.

También faltan:
- límite de bytes
- validación MIME real
- límite de píxeles
- control de acceso
- retención
- eliminación
- manejo de imagen corrupta

A-07. MOTIVO SIGUE SIENDO OPCIONAL

RegistrarEntradaDto usa IsOptional para motivoAcceso.
La UI también lo presenta como opcional.
La base ya contiene 2 registros sin motivo.

Esto impide garantizar el reporte “quién estuvo y por qué”.

A-08. PROVEEDOR Y SERVICIO EXTERNO PIERDEN EMPRESA/CÉDULA

RegistroPage, al seleccionar una persona, envía únicamente personaId y nombre.
No copia cédula ni empresa del resultado seleccionado.

Los reportes de PL, cocina, proveedor y conductor quedan incompletos.

A-09. EVENTOS DE CAPACITACIÓN NO SE FILTRAN POR EDIFICIO Y FECHA

La UI sigue mostrando todos los eventos activos.
Puede asociarse un curso de otro edificio o fuera de fecha.
Seleccionar evento continúa siendo opcional cuando se marca capacitación.

A-10. CERO PRUEBAS UNITARIAS Y E2E INUTILIZABLE

- Unitarias: 0.
- E2E: 0 ejecutadas.
- Error ESM de uuid sin resolver.
- Caso Hello World obsoleto.

Esto permite introducir regresiones como C-02 sin detectarlas.

A-11. AUTORIZACIÓN DE DATOS NO ESTÁ LIMITADA POR EDIFICIO

Todos los roles registrador pueden consultar:
- todos los edificios
- todos los pendientes
- todos los reportes
- empleados y personas externas

EdificioIdDefecto solo preselecciona la UI; no limita el backend.

Debe definirse la política real:

- Si CPF es un puesto por edificio, el JWT debe llevar edificios autorizados.
- El backend debe forzar esos edificios.
- Admin puede consultar todo.
- Un registrador no debe ampliar su alcance enviando otro edificioId.

Este punto es especialmente importante por cédulas, fotos y trazabilidad.

7. HALLAZGOS MEDIOS
======================================================================

M-01. FECHA “HASTA” DEL REPORTE SIGUE EXCLUYENDO EL ÚLTIMO DÍA

ReportsPage envía AAAA-MM-DD.
AccesoService crea new Date(hasta).
SQL usa FechaEntrada <= @Hasta.

El límite queda al inicio del día seleccionado.

Solución:
Usar FechaEntrada < DATEADD(day, 1, @Hasta).

M-02. EXPORTAR CSV SOLO DESCARGA LA PÁGINA VISIBLE

El reporte pagina de 25 en 25 y exportCSV usa el estado data actual.

El usuario puede creer que exportó “el reporte”, cuando solo exportó hasta 25
filas.

Debe:
- pedir todos los datos filtrados a un endpoint de exportación, o
- indicar claramente “Exportar esta página”.

M-03. CSV VULNERABLE A FÓRMULAS DE HOJA DE CÁLCULO

Valores que empiezan con =, +, - o @ pueden ejecutarse como fórmula al abrir el
CSV en Excel.

Los nombres, empresas y motivos pueden contener texto controlado por usuarios.
Sanitizar celdas antes de exportar.

M-04. TIPOS INCOMPLETOS EN FILTRO DE REPORTES

ReportsPage no incluye:
- INSTRUCTOR_INTERNO
- SALIDA_INDEPENDIENTE

Además muestra códigos técnicos en vez de etiquetas sencillas.

M-05. CAMBIAR FILTROS DISPARA CONSULTAS AUTOMÁTICAS

load depende de todos los filtros y useEffect depende de load.
Cada modificación de filtro ejecuta el reporte, aunque existe botón Buscar.

El botón Buscar produce una segunda lógica redundante.

Elegir:
- búsqueda automática con debounce, o
- filtros borrador y botón Aplicar.

M-06. DTO TODAVÍA INCOMPLETOS

Faltan, entre otros:

- Max para porPagina.
- IsIn para tipoPersona en reporte.
- IsDateString para desde/hasta.
- MaxLength para nombres, IDs, empresa, motivo y observación.
- Type(() => Number) e IsInt para IDs CPF.
- validación hasta >= desde.
- evento obligatorio si el acceso es capacitación.
- motivoDetalle obligatorio cuando motivo=Otro.

M-07. CORS SIGUE ABIERTO

main.ts mantiene app.enableCors() sin orígenes permitidos.

M-08. SIN RATE LIMIT, BLOQUEO NI AUDITORÍA DE LOGIN

No se observó Throttler ni política de intentos fallidos.

M-09. SQL TEST SIGUE APUNTANDO A LA BASE EQUIVOCADA POR DEFECTO

nest/.env conserva DB_DATABASE=Asistencia.
test/sql_tests.js toma DB_DATABASE si DB_NAME no existe.
La aplicación, en cambio, usa ControlAcceso escrito en DatabaseService.

El comando simple documentado puede fallar o auditar otro sistema.

M-10. PRUEBAS EXTERNAS ESTÁN DESACTUALIZADAS

No cubren:
- /acceso/pendientes
- /acceso/salida-independiente
- /auth/admin-reset-password
- validaciones DTO nuevas
- página de reportes
- CSV
- alcance por edificio

Además reportan como ausente un endpoint que sí existe.
