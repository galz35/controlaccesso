4. PROBLEMAS ACTUALES PRIORITARIOS
===============================================================================

4.1 P0 — Se puede intentar registrar una persona inexistente
-------------------------------------------------------------

En RegistroPage.tsx, registrarEntrada solo bloquea si no hay edificio. Para un
colaborador, proveedor o instructor, si no se seleccionó un resultado, el código
entra en la rama manual y puede enviar:

- personaId = manual;
- nombrePersona vacío;
- cédula vacía;
- empresa vacía.

Para VISITANTE tampoco se valida nombre antes de enviar.

Esto es un defecto grave de experiencia y prevención de errores.

Regla requerida:

- EMPLEADO/PROVEEDOR/INSTRUCTOR: selected es obligatorio.
- VISITANTE: nombreManual es obligatorio.
- edificioId siempre obligatorio.
- El botón Registrar Entrada debe estar deshabilitado hasta cumplir condiciones.
- Debe mostrarse qué falta junto al paso correspondiente.
- No usar solamente un SweetAlert después del click.

4.2 P0 — Los errores de red desaparecen
----------------------------------------

Hay al menos seis catch vacíos en Dashboard, Registro, SalidaPanel y Catálogos.
Cuando falla la API, el usuario puede ver “sin registros” y creer que no hay nadie
dentro, lo cual es peligroso para control de acceso.

Regla requerida:

- Nunca representar error como cero registros.
- Mantener estados separados: idle, loading, success, empty, error.
- Mostrar “No pudimos consultar los accesos” + botón Reintentar.
- En salida, si no carga la lista, no mostrar “No hay personas pendientes”.
- Registrar errores técnicos en consola/log, pero dar texto simple al usuario.

4.3 P0 — Menú y rutas no se adaptan al rol
-------------------------------------------

README/backend define:

- admin: CRUD, registro, catálogos y administración.
- registrador: entradas/salidas y lectura de catálogos.

Pero el frontend:

- muestra Usuarios CPF a todos;
- permite navegar a admin-cpf solo por estar autenticado;
- muestra Nuevo/Editar catálogos a registrador aunque backend devolverá 403;
- ProtectedRoute valida autenticación, no rol.

Diseño requerido:

- admin ve toda la navegación.
- registrador ve Registro y, si es útil, consultas de catálogos sin botones de alta
  o edición.
- Usuarios CPF solo aparece para admin.
- La ruta admin-cpf debe usar RoleRoute/RequireRole.
- Si alguien abre una URL no autorizada, mostrar página 403 simple y útil.

4.4 P0 — Spinners aplicados incorrectamente a iconos SVG
---------------------------------------------------------

La clase .spinner dibuja un círculo con border. Se aplica también a Loader2 y
RefreshCw, por ejemplo:

- <Loader2 className="icon icon--sm spinner" />
- <RefreshCw className="... spinner" />

Esto combina el dibujo Lucide con bordes de spinner y puede verse deformado.

Solución:

- .spinner se usa en un span/div vacío.
- .icon--spin solo aplica animation: spin.
- Nunca poner ambas clases en el mismo SVG.

4.5 P0 — Iconos pueden apilarse sobre el texto
-----------------------------------------------

reset.css define img, svg { display: block }. Varios títulos insertan un SVG dentro
de un span/h1 sin display flex:

- Accesos de hoy.
- Registrar Entrada.
- Registrar Salida.
- Usuarios Externos.
- Catálogos.

La utilidad vertical-align no corrige un SVG display:block. El resultado puede ser
icono arriba y texto abajo o alineación irregular.

Solución:

- Crear .heading-with-icon o .card-title con inline-flex.
- Hacer .page-header__title inline-flex.
- Aplicar gap y align-items:center.
- No depender de vertical-align.

4.6 P1 — aria-controls apunta a un id inexistente
-------------------------------------------------

El botón menú declara aria-controls="sidebar", pero <aside> no tiene id="sidebar".
Debe corregirse. También faltan focus trap y retorno de foco al cerrar drawer.

4.7 P1 — Solo Login es un formulario real
-------------------------------------------

Registro, Catálogos y Admin CPF usan div + botones onClick. Esto produce:

- Enter inconsistente;
- validación nativa desaprovechada;
- botones sin type;
- semántica débil;
- mayor dificultad para lectores de pantalla.

Todos los bloques de alta/edición/registro deben ser <form onSubmit>.

4.8 P1 — 10 labels no están asociados
---------------------------------------

Existen 18 labels, pero solo 8 tienen htmlFor. Los faltantes se concentran en:

- Admin CPF;
- campos dinámicos de Catálogos;
- tipo/búsqueda/foto en Registro.

Todo campo debe tener id estable y label asociado. Para tipo de persona usar
fieldset + legend, no label suelto sobre un grupo de botones.

4.9 P1 — Pestañas de Login con semántica incompleta
----------------------------------------------------

Se usa role="tablist" y role="tab", pero no hay:

- id de cada tab;
- aria-controls;
- tabpanel;
- navegación con flechas.

Para este caso sencillo es preferible un fieldset con dos opciones segmentadas o
botones con aria-pressed. Evita fingir un patrón ARIA incompleto.

4.10 P1 — Tablas móviles solo tienen scroll horizontal
--------------------------------------------------------

El wrapper evita romper el layout, pero un recepcionista en teléfono debe deslizar
para descubrir nombre, hora y acción. La acción Salida puede quedar fuera de vista.

Solución recomendada:

- Tabla en desktop/tablet ancha.
- Lista de cards operativas en móvil.
- Cada card muestra nombre, tipo, edificio, entrada, estado y acción visible.
- No comprimir todo a texto de 10-11 px.

4.11 P1 — Acción de salida sin confirmación ni estado propio
------------------------------------------------------------

Un click registra la salida inmediatamente. En una lista densa es fácil equivocarse.

Requerido:

- Botón “Registrar salida”.
- Confirmación clara: “¿Registrar salida de María López?”
- Mostrar edificio y hora de entrada.
- Durante envío deshabilitar solo esa fila.
- Éxito verde con nombre y hora.
- Error mantiene la persona en la lista y ofrece Reintentar.

4.12 P1 — Texto y objetivos táctiles pequeños
----------------------------------------------

- .btn--sm puede quedar en 34 px.
- algunos textos usan 11 px.
- KPI labels usan 11 px.
- acciones de tabla son pequeñas para teléfono.

Para usuarios no técnicos y pantallas táctiles:

- acciones principales mínimo 44 px;
- icon-only mínimo 44x44 móvil;
- texto operativo mínimo 13-14 px;
- 11-12 px solo metadata no crítica.

4.13 P1 — SweetAlert no tiene tema central
------------------------------------------

Admin CPF fija rojo solo en un diálogo. Otros SweetAlert pueden usar colores por
defecto. Crear helper/mixin único para confirmar rojo, cancelar gris y éxito verde.

4.14 P2 — Assets prohibidos aunque estén sin uso
-------------------------------------------------

No hay referencias a estos archivos, pero siguen en src/public:

- react.svg con #00D8FF azul.
- vite.svg con identidad violeta/azulada.
- hero.png con base púrpura/violeta.
- icons.svg contiene un símbolo llamado bluesky, aunque el path visible es negro.

La regla corporativa prohíbe azul y púrpura. Después de confirmar que no se usan,
eliminarlos del proyecto/build o reemplazarlos por activos autorizados rojo/blanco.

4.15 P2 — Dependencia remota de Google Fonts
--------------------------------------------

index.html carga Inter/Outfit desde Google. Hay fallbacks correctos, pero una red
corporativa o CSP puede bloquearlas. Para máxima consistencia, autoalojar WOFF2 si
licencia/política lo permite. No es un bloqueo funcional.

4.16 P2 — Detalles CSS/HTML adicionales
----------------------------------------

- .table td--muted, td--center, td--bold y td--small son selectores incorrectos si
  la intención era una clase; debería ser td.td--muted o .td--muted.
- topbar y sidebar duplican marca/usuario/logout en escritorio.
- Dashboard usa <a href> a Registro en vez de Link/NavLink.
- SSO combina login-form y sso-section; .sso-section sobrescribe padding horizontal
  y puede dejar el contenido demasiado pegado.
- No hay captions en tres tablas.
- Catálogo marca todos los campos como requeridos aunque no valida ninguno.
- El tipo “ID Referencia” es lenguaje técnico para usuario no informático.
- “Usuarios CPF” puede ser una sigla desconocida; agregar explicación visible.
- Contador “persona(s) dentro del edificio” es poco natural y puede ser incorrecto
  si lista varias sedes. Usar pluralización real y “en las instalaciones” o filtrar
  por edificio seleccionado.

5. PERSONAS Y NECESIDADES DE USO
===============================================================================

5.1 Recepcionista/registrador
-----------------------------

Conocimiento técnico: bajo.
Frecuencia: alta, muchas operaciones repetidas.
Objetivo: registrar entrada/salida en menos de 20 segundos.
Riesgos: elegir persona equivocada, edificio equivocado o salida accidental.

Debe ver:

- Registro como pantalla inicial.
- Edificio actual claramente visible.
- Entrada y salida, sin módulos administrativos.
- Personas dentro y alertas operativas.

No debe ver:

- Usuarios CPF.
- Botones Nuevo/Editar de catálogos.
- IDs internos.
- errores HTTP, JSON o mensajes técnicos.

5.2 Administrador
-----------------

Conocimiento técnico: medio.
Objetivo: supervisar, mantener catálogos y usuarios.

Debe ver:

- Dashboard.
- Registro.
- Catálogos completos.
- Usuarios externos.
- indicadores y errores de sincronización.

5.3 Usuario externo CPF
-----------------------

Conocimiento técnico: bajo.
Objetivo: iniciar sesión y realizar únicamente las acciones permitidas.

La navegación debe derivarse de su rol efectivo. No asumir que todo CPF puede
administrar o registrar.

5.4 Persona que llega al edificio
---------------------------------

No interactúa siempre con la pantalla, pero espera un proceso rápido. El diseño
debe permitir al recepcionista mantener contacto visual y evitar formularios largos.

6. PRINCIPIOS DEL MEJOR DISEÑO PARA ESTE SISTEMA
===============================================================================

1. Una pantalla, una tarea principal.
2. Mostrar solo lo necesario para el rol.
3. Pedir primero la información que desbloquea la siguiente.
4. No mostrar todos los campos al mismo tiempo si todavía no aplican.
5. Prevenir el error antes del envío.
6. Confirmar acciones irreversibles o fáciles de confundir.
7. Explicar con palabras comunes, no siglas o nombres de base de datos.
8. Nunca convertir un error de red en “no hay datos”.
9. Mantener la acción principal siempre visible.
10. Rojo para marca/acción primaria, carbón para salida, verde solo para resultado.
11. No depender del color: incluir texto e icono.
12. Optimizar velocidad repetitiva sin sacrificar seguridad.

8. CONTEXTO DE EDIFICIO: MEJORA CLAVE
===============================================================================

Si el registrador trabaja normalmente en una sede, pedir edificio en cada entrada
es repetitivo y aumenta errores.

Diseño propuesto:

- Selector “Edificio actual” en la parte superior de Registro.
- Selección obligatoria al comenzar turno/sesión.
- Persistir en sessionStorage o preferencia local, no en backend sin aprobación.
- Mostrarlo siempre en una píldora/bloque: “Registrando en: Edificio Central”.
- Permitir cambiarlo con confirmación.
- Todos los registros nuevos heredan ese edificio.
- Si el sistema debe manejar múltiples edificios por operación, conservar select
  por registro, pero preseleccionarlo con el contexto.

Esto reduce un paso sin ocultar información crítica.

9. FLUJO IDEAL DE LOGIN
===============================================================================

Entrada directa:

1. Mostrar marca Control de Acceso.
2. Preguntar con lenguaje claro:
   - “Soy colaborador Claro”.
   - “Soy proveedor o facilitador”.
3. Mostrar únicamente los campos del modo elegido.
4. Botón rojo “Ingresar”.
5. Mientras valida: “Verificando acceso…”.
6. Si falla: mensaje concreto y próximo paso.

Mensajes recomendados:

- Carnet vacío: “Escriba su número de carnet.”
- Usuario vacío: “Escriba su usuario.”
- Contraseña vacía: “Escriba su contraseña.”
- Credenciales inválidas: “No reconocimos esos datos. Revise e intente nuevamente.”
- Red: “No pudimos conectar con el sistema. Intente de nuevo.”

SSO:

1. “Validando acceso desde el Portal Claro”.
2. Spinner y texto.
3. Si funciona, redirigir.
4. Si falla, explicar: “El enlace venció o no pudo validarse”.
5. Botones: “Volver al Portal” si existe URL autorizada y “Usar otro método”.

No mostrar “token”, “JWT”, “401” ni “SSO” sin explicación al usuario final.

10. FLUJO IDEAL DE REGISTRO DE ENTRADA
===============================================================================

Objetivo de tiempo: 10-20 segundos para persona conocida.

PASO 0 — Confirmar contexto

- Mostrar edificio actual.
- Si no existe, pedir seleccionarlo antes de continuar.

PASO 1 — ¿Quién ingresa?

Opciones grandes:

- Colaborador.
- Proveedor.
- Facilitador externo.
- Facilitador interno.
- Visitante.

No mostrar como cinco botones pequeños. Usar tarjetas segmentadas o grid 2/3 con
texto de 14 px y área táctil 44 px.

PASO 2A — Persona registrada

- Campo “Nombre o carnet”.
- Botón Buscar.
- Permitir Enter.
- Mostrar resultados con nombre grande y carnet/empresa debajo.
- Al elegir, cerrar resultados y mostrar resumen verde suave.
- Acción “Cambiar persona”.
- Si no hay resultados: “No encontramos coincidencias” y orientación según tipo.

PASO 2B — Visitante

- Nombre completo obligatorio.
- Cédula recomendada/obligatoria según política, no inventarla desde frontend.
- Empresa o motivo.
- Labels visibles.

PASO 3 — Datos de la visita

- Edificio preseleccionado por contexto.
- Curso/capacitación opcional.
- Foto opcional con preview y botón Quitar.

PASO 4 — Confirmar

Antes del botón, mostrar resumen compacto:

Persona: Ana Pérez
Tipo: Proveedor
Edificio: Central
Curso: No aplica
Foto: Adjunta / Sin foto

Botón rojo: “Registrar entrada”.
Debe estar disabled mientras falte algo.

PASO 5 — Resultado

- Card/toast verde: “Entrada registrada”.
- Mostrar nombre, edificio y hora.
- Botón “Registrar otra persona”.
- Limpiar persona/visitante/curso/foto.
- Conservar edificio actual para rapidez.

===============================================================================

1. Abrir modo Salida.
2. Mostrar cantidad dentro.

12. DISEÑO IDEAL POR PANTALLA

12.1 Shell
-----------

- Sidebar blanca 270 px en desktop.
- Logo/bloque rojo autorizado.
- Ruta activa rojo suave con barra roja.

- En desktop evitar duplicar nombre y logout en sidebar + topbar.

12.2 Login

El card actual de 400 px es simple y correcto. Para mejorarlo sin complicarlo:

12.3 Dashboard
--------------

Network Error                   No pudimos conectar. Revise e intente de nuevo.

ID Referencia                   Registro relacionado / Seleccione proveedor.
CPF                             Usuario externo (CPF).
persona(s)                      1 persona / 2 personas.

Error                           No se pudo registrar la entrada.

1. Qué ocurrió.
2. Qué información se conservó.

entrada o salida rápida y correctamente, sin necesitar capacitación especial.

- rojo y blanco como marca;
- gris/negro auxiliares;
- verde solo para éxito/estado válido;
- cero azul/cian/teal/índigo/violeta/púrpura;
- rutas, API, SSO y contratos existentes;
- roles admin y registrador del backend.
