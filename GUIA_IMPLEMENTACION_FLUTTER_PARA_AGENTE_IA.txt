0. INSTRUCCIÓN PRINCIPAL PARA EL AGENTE
======================================================================

Construir una aplicación Flutter Android-first, compatible con iOS y adaptable a
teléfono/tablet, para operar el sistema Control de Acceso. La aplicación debe usar
la API NestJS existente, respetar los roles admin/registrador y soportar dos métodos
de identidad:

1. EMPLEADO: SSO corporativo real.
2. USUARIO EXTERNO CPF: usuario + contraseña.

NO implementar login de empleado ingresando solo el carnet.
NO llamar POST /auth/dev-login desde Flutter.
NO guardar JWT ni contraseña en SharedPreferences.
NO inventar endpoints, parámetros SSO, refresh tokens o reglas de negocio.
NO comenzar integración final mientras los bloqueos de la sección 2 sigan abiertos.

El agente debe trabajar por fases pequeñas, compilar y probar después de cada fase.
Debe crear primero la estructura, contratos y pruebas; luego conectar la API real.
Si un contrato no está disponible, usar una interfaz y fake local claramente marcado,
nunca una implementación productiva ficticia.

======================================================================
1. OBJETIVO Y ALCANCE DE LA APP
======================================================================

Objetivo principal
------------------

Permitir a personal autorizado registrar y consultar entradas/salidas de empleados,
proveedores, instructores y visitantes desde un dispositivo móvil o tablet en una
portería, recepción o puesto de seguridad.

Alcance MVP Flutter
-------------------

- Arranque seguro y restauración de sesión.
- Login empleado por SSO.
- Login externo CPF por usuario/contraseña.
- Cierre de sesión y expiración controlada.
- Dashboard de accesos del día.
- Registro de entrada.
- Registro rápido de salida.
- Búsqueda de personas por tipo.
- Selección de edificio y evento de curso.
- Captura/subida opcional de foto cuando backend lo habilite correctamente.
- Catálogos para admin: edificios, proveedores, instructores y cursos.
- Reporte histórico paginado.
- Perfil y cambio de contraseña CPF solo después de corregir el endpoint.
- Estados loading, empty, error, offline y unauthorized diferenciados.

Fuera del MVP
-------------

- Crear un backend dentro de Flutter.
- Acceso directo desde Flutter a SQL Server.
- Sincronización offline de entradas/salidas sin idempotencia del servidor.
- Reconocimiento facial.
- Apertura física de puertas.
- Push notifications.
- Impresión de gafetes.
- Almacenamiento local permanente de cédulas, fotos o históricos completos.

======================================================================

El agente debe leer esta sección antes de generar código. El código del repositorio,

Estado confirmado

  database/004_create_procedures.sql; conteo observado: 0.

- El proceso activo fue iniciado antes de compilar el commit nuevo.
- El backend nuevo pasa TypeScript, pero tiene 317 problemas de lint.
- El README.md sigue describiendo dev-login y quedó obsoleto para autenticación.

BLOQUEO B-01. Migraciones incompletas
------------------------------------

Antes de reiniciar/desplegar el backend nuevo se debe:

1. Tomar backup verificable.
2. Revisar y aplicar database/004_create_procedures.sql.
3. Confirmar que todos los SP esperados existen.
4. Ejecutar pruebas directas de cada SP.
5. Recién entonces reiniciar backend y hacer smoke tests.

Si se reinicia antes, las operaciones migradas a SP pueden fallar masivamente.

BLOQUEO B-02. Sesión CPF rota por JwtStrategy
---------------------------------------------

El token CPF incluye:

  sub, username, nombre, rol, tipo, cpf=true

Pero nest/src/common/jwt.strategy.ts devuelve solamente:

  carnet, nombre, rol

Como consecuencia, req.user.cpf, req.user.username, req.user.tipo y req.user.sub se
pierden. Esto rompe /auth/me CPF y puede enviar UsuarioRegistra undefined al registrar
una entrada. El backend debe preservar ambos tipos de identidad, por ejemplo:

  actorId: string
  authType: EMPLOYEE_SSO | CPF
  carnet?: string
  cpfUserId?: number
  username?: string
  nombre: string
  rol: admin | registrador
  tipo?: PROVEEDOR | INSTRUCTOR_EXTERNO

Flutter no puede corregir este problema. Debe esperar contrato backend estable.

BLOQUEO B-03. React aún no usa SSO real
--------------------------------------

La pestaña "Empleado" parece SSO, pero todavía llama /auth/dev-login con carnet. La
rama de AuthContext que llama /auth/sso-login no está conectada a un redirect/callback
real. Flutter NO debe copiar este comportamiento.

BLOQUEO B-04. SSO móvil no tiene flujo completo definido
-------------------------------------------------------

El backend solo recibe un JWT ya obtenido:

  POST /auth/sso-login { "token": "<token-del-portal>" }

Faltan datos necesarios para móvil:

- URL oficial de autorización del Portal/IdP.
- Si es OIDC/OAuth2, issuer, clientId, scopes y discovery URL.
- Redirect URI Android/iOS autorizado.
- Flujo PKCE o código de un solo uso.
- Logout/revocación SSO.
- Audience e issuer que debe validar backend.
- Forma de renovar sesión.

No colocar un JWT largo en query string/deep link. La solución recomendada es
Authorization Code + PKCE y canje backend. Si el Portal solo soporta JWT handoff,
Seguridad debe definir un mecanismo de un solo uso y corta duración.

BLOQUEO B-05. sp_Login_Empleado es inseguro
-------------------------------------------

El SP nuevo crea un usuario local para cualquier carnet recibido y ya no comprueba
bdplaner.dbo.p_Usuarios. Mientras /auth/dev-login exista, cualquier texto puede crear
un registrador. Se debe:

- eliminar/deshabilitar dev-login en producción;
- hacer que el flujo SSO sea la única entrada de empleados;
- validar empleado activo y claims del IdP;
- no autoaprovisionar permisos sin política.

BLOQUEO B-06. Cambio de contraseña CPF incompleto
------------------------------------------------

- targetUser se calcula pero no se usa.
- Debido a B-02, req.user.cpf nunca llega correctamente.
- El endpoint permite enviar username arbitrario y siempre exige oldPassword.
- El admin no tiene un flujo separado de reset.
- Se lanza Error genérico en lugar de ForbiddenException.

No mostrar pantalla de cambio/reset en producción hasta definir:

- "cambiar mi contraseña" usando identidad del token, sin username enviado;
- "reset administrativo" separado, auditado y con contraseña temporal/expiración;
- política de contraseña y rate limit.

BLOQUEO B-07. Fotos todavía no tienen contrato operativo completo
-----------------------------------------------------------------

El backend guarda WebP y retorna una ruta, pero Nginx reenvía la ruta al backend y
NestJS no expone el directorio. Además faltan validaciones de archivo y privacidad.
La UI puede preparar el selector/cámara detrás de feature flag, pero no debe depender
de fotos hasta confirmar subida y lectura extremo a extremo.

BLOQUEO B-08. Seguridad y validaciones pendientes
-------------------------------------------------

- DTOs siguen siendo any; ValidationPipe no valida contratos reales.
- Sin rate limiting para contraseña o SSO.
- JWT no tiene refresh endpoint.
- CORS es abierto.
- Secretos históricos continúan rastreados porque .gitignore no elimina archivos ya
  versionados.
- La cuenta SQL observada anteriormente era sa/sysadmin.

Precondición de integración Flutter
-----------------------------------

No declarar integración terminada hasta que un smoke test real confirme:

[ ] POST /auth/sso-login deja de responder 404.
[ ] POST /auth/cpf-login deja de responder 404.
[ ] Los SPs están aplicados y versionados.
[ ] Un usuario CPF puede llamar GET /auth/me correctamente.
[ ] Un usuario CPF puede registrar entrada y UsuarioRegistra no es null.
[ ] Un empleado SSO válido puede llamar GET /auth/me.
[ ] dev-login está deshabilitado en producción.
[ ] 401, 403, 400 y 500 están diferenciados.

Eso NO está soportado por el backend actual. Se requiere un nuevo contrato de
autenticación corporativa. No reutilizar tblUsuariosCPF para empleados sin decisión
formal de Seguridad/RRHH.

9. SESIÓN, JWT Y SEGURIDAD EN FLUTTER
======================================================================

Almacenamiento
--------------

- Guardar access token en flutter_secure_storage.
- Guardar solo metadata mínima de usuario si mejora el splash.
- Nunca guardar password, token SSO de origen ni respuesta completa de búsqueda.
- Borrar secure storage al logout, 401 definitivo o sesión inválida.

Interceptor Dio
---------------

- Añadir Authorization: Bearer únicamente a rutas protegidas.
- No añadir token a login.
- Un solo manejador de 401 para evitar múltiples redirecciones simultáneas.
- No reintentar automáticamente POST de entrada/salida.
- Reintentar GET solo con política limitada y backoff.
- Sanitizar logs: nunca Authorization, password, cédula, foto o payload SSO.

Expiración
----------

El backend no expone refresh token. Por tanto:

- leer exp del JWT solo para UX, no como validación de seguridad;
- si exp pasó, borrar sesión y solicitar login;
- si API retorna 401, volver a login;
- no inventar refresh contra /auth/sso-login;
- cuando backend añada refresh, implementar single-flight para evitar carreras.

SSO seguro
----------

- Preferir navegador del sistema, no WebView embebido.
- Usar PKCE, state y nonce si OIDC.
- Validar scheme/host/path del callback.
- No aceptar token desde cualquier deep link.
- Usar universal/app links verificados en producción.
- No loguear callback URI completo.

10. CLIENTE HTTP Y MANEJO DE ERRORES
======================================================================

Timeouts iniciales sugeridos
----------------------------

  connect: 10 segundos
  receive GET: 20 segundos
  upload foto: 30 segundos

Mapear DioException a ApiFailure tipado:

  validation
  unauthorized
  forbidden
  notFound
  conflict
  rateLimited
  networkUnavailable
  timeout
  server
  unknown

Mensajes UI
-----------

- 400: mostrar validación del servidor de forma segura.
- 401 en login: "Credenciales incorrectas".
- 401 autenticado: "Tu sesión expiró" y volver a login.
- 403: "No tienes permiso para esta acción".
- 404 salida: "El registro ya fue cerrado o no existe".
- 409 duplicado futuro: ofrecer refrescar, no repetir silenciosamente.
- 429: mostrar espera y bloquear temporalmente botón.
- 500/503: "Servicio no disponible. Intenta nuevamente".
- sin red: mantener formulario y permitir reintentar manualmente.

No mostrar stack traces, nombres de SP, errores SQL ni HTML de Nginx.

    -> no hay token -> Login
    -> token presente -> GET /auth/me
       -> 200 -> Home
       -> 401 -> limpiar -> Login
       -> red caída -> Error de conexión + Reintentar

Login:

  Elegir identidad
    -> Empleado -> SSO corporativo -> callback -> sesión -> Home
    -> Externo  -> usuario/contraseña -> sesión -> Home

Home:

  Dashboard
    -> Registrar entrada
    -> Registrar salida
    -> Accesos de hoy
    -> Reportes
    -> Administración, solo admin
    -> Perfil / cerrar sesión

Navegación por dispositivo
--------------------------

Teléfono:

- NavigationBar inferior con máximo 4 destinos:
  Inicio, Entrada, Salida, Más.
- "Más" contiene Reportes, Catálogos, Perfil y Logout según rol.

Tablet:

- NavigationRail o drawer persistente.
- Contenido maestro/detalle cuando sea útil.
- No aumentar simplemente espacios; aprovechar ancho para filtros + resultados.

Guards de ruta
--------------

- Sin sesión: solo login/callback.
- registrador: inicio, entrada, salida, hoy, reporte permitido por API.
- admin: además catálogos y usuarios CPF cuando se habiliten.
- unknown role: pantalla sin permisos y logout.

La UI oculta acciones no permitidas, pero backend sigue siendo autoridad.

15.1 Splash / Session Bootstrap

15.2 Login

Cabecera:

Empleado:

Encabezado:

- refresh y estado de última actualización.

KPIs:

- Entradas hoy.
- Salidas hoy.
- Personas dentro.

Acciones grandes:

- Registrar salida, azul.

Lista reciente:

- pull-to-refresh.
- actualización cada 30-60 s solo mientras pantalla está visible, hasta disponer de
  tiempo real.

15.4 Registrar entrada
----------------------

Usar formulario progresivo en una sola pantalla, no wizard largo. Una portería debe
registrar con pocos toques.

Orden:

1. Selector de tipo de persona.
2. Búsqueda o campos de visitante.
3. Tarjeta de persona seleccionada.
4. Edificio; preseleccionar solo si backend asigna scope al operador.
5. Evento de curso opcional filtrado por edificio y fecha.
6. Foto opcional detrás de feature flag.
7. Resumen compacto.
8. Botón fijo "Registrar entrada".

Por tipo:

- EMPLEADO/INSTRUCTOR_INTERNO: /search/empleado.
- PROVEEDOR: /search/proveedor.
- INSTRUCTOR_EXTERNO: /search/instructor.
- VISITANTE: nombre, cédula/documento, empresa/motivo según regla aprobada.

Después de éxito:

- mostrar confirmación con nombre, edificio y hora;
- limpiar persona/evento/foto;
- mantener edificio solo si puesto fijo y usuario lo desea;
- ofrecer "Registrar otra" o "Ir al inicio".

15.5 Registrar salida
---------------------

- Cargar /acceso/hoy.
- Filtrar fechaSalida == null.
- Filtro por edificio y búsqueda local por nombre/documento en resultados cargados.
- Mostrar hora de entrada y tiempo dentro.
- Tap abre bottom sheet de confirmación.
- POST /acceso/salida/:id una sola vez.
- En éxito retirar elemento y actualizar dashboard.
- En 404 refrescar y explicar que otro puesto ya registró la salida.

15.6 Accesos de hoy
-------------------

- Filtro edificio.
- Chips Todos / Dentro / Salieron.
- Lista ordenada por entrada descendente.
- Detalle con PII mínima.
- Pull-to-refresh y estado vacío real.

15.7 Reportes
-------------

- Desde/hasta.
- Edificio.
- Tipo de persona.
- Paginación/infinite scroll controlado.
- Indicador total.

Advertencia actual: "hasta" se interpreta como instante de medianoche si se envía
solo fecha. Hasta que backend implemente rango exclusivo, Flutter puede enviar fin
del día local cuidadosamente, pero debe preferirse corregir el contrato backend.

15.8 Catálogos admin
--------------------

- Lista + búsqueda.
- FAB Agregar.
- Formulario Crear/Editar.
- Confirmar desactivación.

15.9 Perfil

Riverpod recomendado

Invalidaciones después de mutaciones
------------------------------------

Entrada exitosa:

- invalidar accesos de hoy;
- invalidar KPIs/dashboard;
- no invalidar todos los catálogos.

Salida exitosa:

- actualizar optimistamente solo después de 200 o usar invalidación inmediata;
- invalidar accesos de hoy y KPIs.

No mantener dos fuentes de verdad independientes para la misma lista.

Offline
-------

MVP: lectura de último estado solo si privacidad lo permite; mutaciones deshabilitadas
sin red. No crear cola de entradas/salidas porque backend no tiene idempotency key y
podría duplicar registros al reconectar.
