   en database/006_migration_v3.sql ni en ningún otro script versionado. El Git
   no puede reconstruir lo que está ejecutándose.

5. El login por carnet continúa visible en producción, pero el endpoint
   dev-login ahora responde 403 correctamente. Para un empleado que entre por
   la página de login, la interfaz ofrece una opción que siempre falla.

6. Las fotos se publican sin JWT bajo /control-acceso-uploads/. Un UUID difícil

4. MODELO FUNCIONAL CORRECTO PARA CLARO
======================================================================

Actores:

1. CPF/registrador:
   Opera un puesto físico. Su trabajo principal es buscar o escanear una
   persona, confirmar edificio y registrar el movimiento.

2. Administrador:
   Gestiona catálogos, CPF, edificios, reportes globales y correcciones.

3. Empleado:
   Se identifica por carnet físico/código. No marca asistencia laboral.

4. Visitante:
   Puede registrarse manualmente. Nombre y motivo obligatorios; cédula y empresa
   opcionales según política.

5. Proveedor/instructor:
   Se busca en catálogo y conserva cédula/empresa.

6. Personal externo:
   Incluye PL, cocina, carga/descarga, conductor, mantenimiento y servicios.

7. Capacitación:
   Es una característica de un solo edificio. Entrar a ese edificio no significa
   necesariamente asistir a capacitación.

Flujo de entrada recomendado:

1. El CPF inicia sesión.
2. Su edificio aparece fijo o preseleccionado.
3. Escanea carnet/código o busca nombre.
4. El sistema detecta tipo cuando sea posible.
5. Muestra una tarjeta grande con nombre, código, empresa y cédula.
6. CPF elige un motivo sencillo.
7. Solo si está en el edificio de capacitación pregunta:
   “¿Viene a una capacitación?” Sí/No.
8. Si responde Sí, curso/evento es obligatorio.
9. Si responde No, no se muestra curso.
10. CPF confirma “Registrar entrada”.
11. Confirmación verde durante 1-2 segundos.
12. El formulario vuelve al campo de escaneo sin perder el edificio.

Flujo de salida normal:

1. CPF abre “Salidas”.
2. Busca entre todas las entradas abiertas, no solo las de hoy.
3. Confirma persona, edificio y hora de entrada.
4. Registra salida.
5. El registro deja de aparecer entre pendientes.

La salida es opcional:
No debe obligarse a cerrar todos los movimientos ni interpretar una entrada
abierta como incumplimiento laboral. “Sin salida registrada” describe mejor el
estado que “Dentro” cuando el sistema admite omisiones.

Flujo de salida sin entrada:

1. La acción excepcional siempre está visible.
2. CPF selecciona/usa su edificio autorizado.
3. Busca carnet/código o escribe los datos mínimos.
4. Escribe observación obligatoria.
5. El sistema muestra:
   “No se encontró una entrada abierta. Se guardará solamente la novedad de
   salida.”
6. CPF confirma.
7. Se registra una salida/novedad, nunca una persona pendiente.

Reporte:

Debe responder:
- quién;
- tipo de persona;
- código/carnet/cédula;
- empresa;
- edificio;
- motivo y detalle;
- fecha/hora de entrada, si existe;
- fecha/hora de salida, si existe;
- tipo de movimiento;
- CPF que registró;
- si fue capacitación y qué evento.

5. HALLAZGOS CRÍTICOS
======================================================================

C-01. AUTORIZACIÓN POR EDIFICIO NO ESTÁ ACTIVA

Evidencia:
- nest/src/common/building.guard.ts existe.
- No hay ninguna referencia de uso a BuildingGuard.
- AccesoController solo aplica AuthGuard y RolesGuard.
- CpfAuthService no firma edificioIdDefecto en el JWT.
- JwtStrategy no devuelve edificioIdDefecto.
- BuildingGuard permite continuar si el claim no existe.
- /acceso/pendientes y /acceso/reporte aceptan edificio opcional; sin edificio
  devuelven todos.

Impacto:
Un CPF de un edificio puede consultar datos, cédulas, motivos, fotografías y
pendientes de otros edificios. También puede registrar una entrada o salida
contra un edificio distinto enviando otro ID.

Por qué el guard actual no basta:
Aunque se añadiera @UseGuards(BuildingGuard), una petición sin edificioId pasa.
Después el servicio consulta todos los edificios. El alcance debe forzarse, no
solo compararse cuando el cliente decide enviar el filtro.

Corrección mínima:
Ver parche P0-02 en la sección 17.

Criterio de aceptación:
- CPF con edificio 1 y query edificio 2 recibe 403.
- CPF con edificio 1 y sin query recibe únicamente edificio 1.
- Admin puede consultar todos.
- CPF sin edificio configurado recibe 403 operativo, no acceso global.

C-02. RATE LIMITER PUEDE BLOQUEAR A TODOS LOS USUARIOS

Evidencia:
- RateLimiterMiddleware usa req.ip antes que X-Forwarded-For.
- main.ts no contiene app.set('trust proxy', 1).
- Nginx reenvía al backend por 127.0.0.1.
- Middleware se aplica a forRoutes('*').
- Límite único: 60 solicitudes/minuto por clave.

Una sola pantalla puede producir varias solicitudes:
- /auth/me;
- /edificios;
- /eventos-curso;
- /acceso/pendientes;
- búsquedas;
- reporte y catálogos.

Con varios CPF, 60/minuto es bajo incluso si la IP se resolviera correctamente.
Sin trust proxy, todos probablemente comparten 127.0.0.1.

Impacto:
Denegación de servicio accidental. Un usuario activo o un script puede provocar
HTTP 429 para todo el personal.

Problemas secundarios:
- almacén en memoria no sirve con múltiples procesos;
- reiniciar borra contadores;
- el setInterval no usa unref();
- E2E/Jest queda abierto;
- no diferencia login sensible de lecturas normales.

Corrección mínima:
Ver P0-03.

C-03. SALIDA SIN ENTRADA NO FUNCIONA DESDE REACT

Evidencia:
- React guarda ssForm.nombre.
- React envía {...ssForm, edificioId: 1}.
- DTO exige nombrePersona.
- ValidationPipe usa forbidNonWhitelisted.
- Prueba exacta: HTTP 400.

Además:
- El botón solo se renderiza cuando filtrados.length > 0.
- Si no hay pendientes, la acción desaparece justamente cuando puede necesitarse.
- El edificio está escrito como 1.
- El campo motivoAcceso declarado en el servicio no se utiliza.
- El backend responde tipo: SALIDA_NOCTROL, texto incorrecto.

Impacto:
Una de las reglas expresamente solicitadas por negocio está rota.

Corrección mínima:
Ver P0-01.

C-04. BASE ACTIVA Y GIT NO SON REPRODUCIBLES

Evidencia:
- La base activa contiene sp_Acceso_Pendientes.
- La base activa contiene sp_Acceso_SalidaIndependiente.
- rg sobre database/ no encuentra su definición.
- database/006_migration_v3.sql solo actualiza entrada, reporte y administración
  CPF.
- tblPersonalExterno tampoco se crea en los scripts base revisados.
- 006 usa u.Correo, pero 004_create_cpf_users.sql no crea esa columna.

Contradicción adicional de tamaños:
- columnas reales:
  MotivoAcceso VARCHAR(50)
  MotivoDetalle VARCHAR(250)
- SP activo y 006:
  @MotivoAcceso VARCHAR(100)
  @MotivoDetalle VARCHAR(500)

Resultado:
Un valor aceptado por DTO/SP puede fallar por truncamiento al insertarse.

Impacto:
Recuperación, despliegue nuevo o ambiente QA pueden compilar y fallar en tiempo
de ejecución. No existe una fuente única de verdad.

Corrección:
Crear database/007_reconciliacion_v4.sql idempotente, probar 001-007 sobre una
base vacía temporal y actualizar los tests para exigir los SP.

6. HALLAZGOS ALTOS
======================================================================

A-01. LOGIN DE EMPLEADO ES UNA RUTA MUERTA EN PRODUCCIÓN

LoginPage muestra la pestaña “Empleado”, pide carnet y AuthContext llama
/auth/dev-login. NODE_ENV=production hace que siempre responda 403.

La seguridad mejoró, pero la experiencia quedó rota.

Solución:
- ocultar esa pestaña en producción;
- mostrar “Entrar desde Portal Claro”;
- recibir al empleado exclusivamente por /auth/sso;
- mantener dev-login solo detrás de VITE_ALLOW_DEV_LOGIN=true y
  ALLOW_DEV_LOGIN=true en entornos no productivos.

A-02. EDIFICIO CPF SE PIERDE AL RECARGAR

CpfAuthService devuelve edificioIdDefecto durante login, pero no lo firma en el
JWT. AuthContext ejecuta /auth/me al recargar y JwtStrategy no conserva ese
campo. Después de F5, RegistroPage ya no puede preseleccionar el edificio.

Solución:
Firmar y validar el edificio, o mejor consultar el CPF activo en base dentro de
/auth/me y devolver permisos vigentes.

A-03. CUENTA CPF DESACTIVADA CONSERVA JWT HASTA EXPIRAR

JwtStrategy confía completamente en claims. Desactivar el usuario cambia SQL,
pero un token emitido sigue válido hasta 8 horas.

Solución rápida:
- agregar tokenVersion o sessionVersion;
- consultar estado activo para operaciones sensibles;
- reducir TTL CPF;
- registrar FechaDesactivacion;
- invalidar sesiones por versión.

A-04. FOTOGRAFÍAS PÚBLICAS

main.ts usa useStaticAssets sin AuthGuard.
Nginx publica /control-acceso-uploads/ con Cache-Control public, immutable.

Riesgos:
- dato personal visible sin autenticación;
- cache público por 30 días;
- no hay restricción por edificio;
- no hay retención/eliminación documentada.

Además, FileInterceptor recibe el archivo completo en memoria antes de que
savePhoto compruebe 5 MB. Nginx permite 20 MB.

Solución:
- límite Multer de 5 MB antes del buffer;
- controlador autenticado de fotos;
- autorización por registro/edificio;
- Cache-Control private, no-store o política aprobada;
- retención automática y registro de acceso.

A-05. SALIDA INDEPENDIENTE SIGUE USANDO UN MODELO AMBIGUO

La base contiene 3 movimientos SALIDA_INDEPENDIENTE. Dos tienen
FechaEntrada=FechaSalida. Eso evita que aparezcan pendientes, pero inventa una
entrada a la misma hora.

Uno de los tres no cumple esa igualdad, lo que corresponde al comportamiento
antiguo o a un estado distinto que debe revisarse manualmente.

No se consultaron datos personales.

Solución corta compatible:
- marcar TipoPersona/TipoMovimiento;
- entrada y salida iguales;
- excluir SALIDA_INDEPENDIENTE de pendientes y KPI de ocupación.

Solución correcta:
Tabla de movimientos/novedades con TipoMovimiento ENTRADA, SALIDA o NOVEDAD y
FechaMovimiento única. No inventar FechaEntrada.

A-06. EXPORTACIÓN DE UN MILLÓN DE FILAS

ReportsPage pide porPagina=1000000.
ReporteQueryDto no tiene @Max.
SQL usa FETCH NEXT con ese valor.

Riesgo:
- memoria alta en SQL, Nest y navegador;
- respuesta lenta;
- fácil abuso autenticado;
- CSV puede congelar equipos modestos.

Solución:
- @Max(500) para paginación normal;
- endpoint /acceso/reporte/export con streaming;
- o iterar páginas de 500/1000 y mostrar progreso.

A-07. CSV NO ESCAPA COMILLAS

Cada celda se rodea con comillas, pero las comillas internas no se duplican.
Un nombre como Empresa "Central" rompe el CSV.

La mitigación de fórmulas con tabulador es parcial. Es preferible anteponer una
comilla simple y escapar comillas dobles.

A-08. DTO Y CONTROLADORES TODAVÍA USAN any

Eventos, edificios, proveedores, instructores, cursos y personal externo usan
@Body() dto: any. Con whitelist global, un objeto sin clase DTO no obtiene una
validación útil de forma/tamaños/reglas.

Admin change-building también usa una interface TypeScript que desaparece en
runtime.

Riesgos:
- campos vacíos;
- strings demasiado largos;
- números como texto;
- errores SQL 500;
- contratos distintos entre web, Flutter y backend.

A-09. CAMBIO DE CONTRASEÑA PROPIO CONFÍA EN username DEL BODY

/auth/cpf-password debería operar sobre req.user.username, no permitir que el
cliente elija la cuenta. Aunque exige contraseña anterior, el contrato es
innecesariamente amplio y facilita errores.

A-10. HEALTH DEVUELVE HTTP 200 AUNQUE LA BASE FALLE

AppController captura el error y devuelve status:error, pero no cambia el código
HTTP. Un monitor que solo verifica HTTP puede considerar saludable una API sin
base.

Usar 503 Service Unavailable cuando SQL no responde.

M-01. CAPACITACIÓN NO ES UNA REGLA OBLIGATORIA

  export function resolveBuilding(

  CREATE OR ALTER PROCEDURE dbo.sp_Acceso_Pendientes
    @EdificioId INT = NULL
  AS
  BEGIN
    SET NOCOUNT ON;

    SELECT
      r.*,
      e.Nombre AS EdificioNombre,
      DATEDIFF(HOUR, r.FechaEntrada, GETDATE()) AS AntiguedadHoras
    FROM dbo.tblRegistroAcceso r
    INNER JOIN dbo.tblEdificios e ON e.Id = r.EdificioId
    WHERE r.FechaSalida IS NULL
      AND r.TipoPersona <> 'SALIDA_INDEPENDIENTE'
      AND (@EdificioId IS NULL OR r.EdificioId = @EdificioId)
    ORDER BY r.FechaEntrada ASC;
  END
  GO
