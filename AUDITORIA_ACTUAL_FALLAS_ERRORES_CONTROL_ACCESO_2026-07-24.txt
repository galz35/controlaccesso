4. HALLAZGOS CRÍTICOS
======================================================================

C-01. DEV-LOGIN ESTÁ ABIERTO EN EL SISTEMA DESPLEGADO

Evidencia:
- nest/src/auth/auth.controller.ts, líneas 17-20.
- react/src/pages/LoginPage.tsx, líneas 20-22.
- react/src/context/AuthContext.tsx, líneas 40-42.

Problema:
POST /api/auth/dev-login no tiene guard, contraseña ni comprobación SSO.
El stored procedure sp_Login_Empleado crea un usuario local si no existe.

Impacto:
Una persona que conozca o adivine un carnet puede obtener un JWT.
El endpoint señalado como “temporal para desarrollo” está publicado.

Corrección obligatoria:

1. Deshabilitar dev-login cuando NODE_ENV=production.
2. Eliminar la pestaña de ingreso por carnet simple en producción.
3. Usar únicamente SSO corporativo para empleados.
4. Validar que el carnet exista y esté activo en la fuente corporativa.
5. No crear usuarios locales a partir de cualquier texto recibido.

C-02. EL JWT CPF PIERDE SU IDENTIDAD AL PASAR POR PASSPORT

Evidencia:
- nest/src/auth/cpf-auth.service.ts, líneas 51-54.
- nest/src/common/jwt.strategy.ts, líneas 14-15.
- nest/src/auth/auth.controller.ts, líneas 58-62.
- nest/src/acceso/acceso.controller.ts, línea 17.

Problema:
CpfAuthService firma sub, username, tipo y cpf=true. JwtStrategy devuelve solo
carnet, nombre y rol. Se pierden cpf, username, sub y tipo.

Impacto:

- /auth/me no reconoce correctamente al usuario CPF.
- El registro de acceso recibe req.user.carnet vacío para un CPF.
- UsuarioRegistra puede llegar nulo al stored procedure.
- Se pierde trazabilidad de qué CPF registró la entrada.
- Las reglas específicas de CPF no pueden aplicarse de forma confiable.

Corrección obligatoria:
JwtStrategy debe conservar explícitamente:

- sub
- carnet
- username
- nombre
- rol
- tipo
- cpf

También debe existir una prueba E2E de login CPF -> /auth/me -> entrada.

C-03. CREDENCIALES Y SECRETOS INSEGUROS

Evidencia:
- test/README.md contiene una contraseña de base escrita directamente.
- El mismo README incluye un carnet concreto.
- nest/src/common/jwt.strategy.ts usa un secreto JWT predeterminado conocido.
- nest/src/auth/auth.module.ts usa el mismo secreto predeterminado.
- README principal documenta secretos de ejemplo demasiado parecidos al uso real.

Impacto:

- La contraseña ya quedó en el historial Git publicado.
- Borrar solo la línea no elimina la exposición histórica.
- Si JWT_SECRET falta, cualquiera que conozca el valor predeterminado puede
  fabricar tokens.

Corrección obligatoria:

1. Rotar la contraseña expuesta.
2. Eliminar credenciales reales de archivos versionados.
3. Considerar limpieza del historial si el repositorio no debe conservarlas.
4. Hacer obligatorio JWT_SECRET y fallar al iniciar si falta.
5. Usar secretos largos y diferentes para JWT de la aplicación y SSO.
6. Añadir escaneo automático de secretos en CI.

C-04. NO EXISTE VALIDACIÓN REAL DE LOS CUERPOS HTTP

Evidencia:
- nest/src/main.ts configura ValidationPipe.
- Controladores y servicios reciben dto: any o tipos TypeScript sin decoradores.
- No hay clases DTO con class-validator.

Problema:
ValidationPipe no puede validar requisitos de runtime usando interfaces o any.

Impacto:

- Se puede enviar EdificioId vacío.
- TipoPersona acepta valores arbitrarios.
- Nombres, carnets, IDs y fechas no tienen límites confiables.
- Paginación acepta valores inválidos.
- Errores llegan a SQL y se convierten en 500.
- Aumenta el riesgo de abuso, datos basura y fallos operativos.

Corrección obligatoria:
Crear DTO reales con IsInt, Min, Max, IsIn, IsString, Length, IsOptional,
IsDateString y validaciones cruzadas.

5. HALLAZGOS ALTOS
======================================================================

A-01. CAMBIO DE CONTRASEÑA ADMINISTRATIVO NO FUNCIONA

Evidencia:
- react/src/pages/AdminCpfPage.tsx, línea 44.
- nest/src/auth/auth.controller.ts, líneas 47-53.
- nest/src/auth/cpf-auth.service.ts, líneas 62-79.

El frontend administrador envía oldPassword vacío. El backend siempre compara
oldPassword contra el hash actual. Por tanto, el botón “Cambiar contraseña”
debe fallar.

Debe separarse:

- Cambio propio: exige contraseña anterior.
- Restablecimiento administrativo: no exige contraseña anterior, pero registra
  auditoría, obliga cambio posterior y requiere rol admin.

A-02. NO SE PUEDE CERRAR DESDE LA UI UNA ENTRADA DE AYER

Evidencia:
- SalidaPanel usa exclusivamente GET /acceso/hoy.
- sp_Acceso_Hoy filtra la fecha actual.

Si una salida es opcional y CPF decide registrarla al día siguiente, la persona
ya no aparece en la pantalla.

Debe existir:
GET /acceso/pendientes?edificioId=...
que devuelva todas las entradas abiertas, con filtros y antigüedad visible.

A-03. NO EXISTE SALIDA SIN ENTRADA PREVIA

El requisito indicado permite registrar salida aunque el papel no tenga entrada
o aunque la entrada se haya omitido. Actualmente solo existe:
POST /acceso/salida/:id

Debe definirse un movimiento o novedad independiente con:

- edificio
- persona/carnet
- fecha y hora
- usuario CPF
- motivo “salida sin entrada encontrada”
- observación obligatoria

No conviene falsificar una entrada automática sin distinguirla.

A-04. NO HAY PANTALLA DE REPORTES

El backend tiene GET /acceso/reporte, pero React no tiene:

- ruta de reportes
- opción en el menú
- filtros
- exportación
- vista “quién estuvo, dónde y por qué”

El Dashboard solo muestra accesos del día y tampoco muestra MotivoAcceso.

A-05. LOS DATOS DE PROVEEDOR/PL SE PIERDEN AL REGISTRAR

Evidencia:
- react/src/pages/RegistroPage.tsx, líneas 90-98.

Cuando se selecciona una persona, solo se envían personaId y nombrePersona.
No se envían cedulaPersona ni empresaPersona del resultado seleccionado.

Impacto:
Los reportes de proveedor, cocinero, conductor o PL quedan incompletos.

A-06. FOTOGRAFÍAS SIN PUBLICACIÓN Y VALIDACIÓN COMPLETAS

Evidencia:
- AccesoService guarda en UPLOAD_PATH/fotos_acceso.
- Retorna /control-acceso-uploads/fotos_acceso/archivo.webp.
- Nginx envía esa ruta al puerto 3001.
- Nest no configura useStaticAssets ni ServeStaticModule.

Resultado probable:
La fotografía puede guardarse, pero la URL pública no tiene una ruta estática
implementada por Nest.

Además faltan:

- límite de tamaño
- límite de dimensiones/píxeles
- validación MIME real
- manejo de archivo corrupto
- política de retención
- autorización de lectura
- privacidad y consentimiento

A-07. SCRIPTS SQL NO RECONSTRUYEN LA BASE ACTUAL

Los scripts versionados no contienen varias columnas y parámetros que sí están
en la base activa. Una instalación nueva o recuperación ante desastre quedaría
incompatible con Nest.

Debe crearse una migración idempotente que incluya:

- tblPersonalExterno
- EsCapacitacion
- EdificioIdDefecto y su FK
- MotivoAcceso
- MotivoDetalle
- stored procedures actualizados
- permisos del usuario de aplicación

A-08. MOTIVO DE ACCESO NO ES OBLIGATORIO

El objetivo del reporte es saber quién estuvo y por qué. La interfaz marca el
motivo como opcional y la base ya tiene 2 registros sin motivo.

Recomendación:

- Hacer MotivoAcceso obligatorio.
- Si se elige “Otro”, hacer MotivoDetalle obligatorio.
- Para salida sin entrada, exigir una observación.
- No usar “Empresa / Motivo” como un único campo ambiguo.

A-09. EVENTO DE CAPACITACIÓN PUEDE NO CORRESPONDER AL EDIFICIO

La interfaz muestra todos los eventos activos, sin filtrar por:

- edificio seleccionado
- fecha vigente
- rango de fechas
- disponibilidad

Además, si se elige “Capacitación”, seleccionar un evento sigue siendo opcional.

Debe filtrar eventos por edificio y fecha, y exigir evento solo cuando el motivo
real sea capacitación.

6. HALLAZGOS MEDIOS
======================================================================

M-01. FECHA “HASTA” DEL REPORTE PUEDE EXCLUIR CASI TODO EL ÚLTIMO DÍA

new Date('AAAA-MM-DD') representa el inicio del día. SQL usa FechaEntrada <=
Hasta. Por tanto, puede excluir registros posteriores a las 00:00.

Solución:
Usar rango semiabierto: FechaEntrada >= Desde y FechaEntrada < día siguiente.

M-02. PAGINACIÓN SIN LÍMITES

pagina y porPagina se convierten con parseInt sin validar:

- cero
- negativos
- NaN
- tamaños excesivos

Limitar porPagina, por ejemplo entre 1 y 100.

M-03. CORS ABIERTO

app.enableCors() se ejecuta sin lista de orígenes.
Restringir al dominio corporativo y métodos necesarios.

M-04. SIN RATE LIMIT EN LOGIN Y BÚSQUEDAS

No hay protección visible contra:

- fuerza bruta CPF
- enumeración de usuarios
- abuso de dev-login
- consultas masivas

Añadir throttling, bloqueo temporal y auditoría de intentos.

M-05. SESIÓN FRONTEND CONFÍA EN localStorage SIN REVALIDAR

AuthProvider considera autenticado al usuario si existe JSON local, incluso si:

- no existe token
- el token expiró
- el rol cambió
- la cuenta fue desactivada

Al cargar, debe consultar /auth/me y mostrar estado de verificación.

M-06. TOKEN SSO EN QUERY STRING

SsoHandlerPage toma token de window.location.search. Puede filtrarse a:

- historial
- logs del proxy
- encabezado Referer
- capturas

Preferir código de un solo uso, fragmento URL o intercambio backend seguro.
Limpiar la URL inmediatamente con history.replaceState.

M-07. DOMINIO DE FOTOS ESCRITO EN EL COMPONENTE

Dashboard concatena https://rhclaroni.com directamente.
Esto falla o mezcla entornos de desarrollo, prueba y producción.

Usar URL relativa o variable de entorno.

M-08. ERRORES SQL NO SE TRADUCEN A MENSAJES HTTP ÚTILES

Duplicados y datos incompletos aparecen como RequestError/500 en logs.
Debe responder 400 o 409 con mensaje controlado y correlation ID.

M-09. ADMINISTRACIÓN CPF INCOMPLETA

No existe interfaz/API visible para:

- desactivar cuenta
- reactivar
- cambiar edificio por defecto
- cambiar tipo
- ver último acceso
- obligar cambio de contraseña
- revisar auditoría

M-10. FLUJO DE CARNET NO ESTÁ OPTIMIZADO PARA ESCÁNER

El campo permite escribir y Enter, pero no hay:

- modo escáner explícito
- prevención clara del submit accidental
- selección automática por coincidencia exacta
- confirmación visual grande
- foco restaurado después de guardar
- sonido/feedback para puesto CPF

Para reemplazar papel, el flujo ideal debe poder operar casi sin mouse.
