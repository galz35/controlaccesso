Control Acceso ya es técnicamente superior a Asistencia como base frontend.

No conviene clonarlo nuevamente ni reemplazar su CSS con el CSS de Asistencia.

- organización CSS;

- reducción de estilos inline;
- asociación de labels;
- formularios semánticos;

- confirmar y bloquear correctamente la salida;

5. PENDIENTES OBLIGATORIOS — PRIORIDAD ALTA
===============================================================================

5.1 Salida sin confirmación
---------------------------

Archivo:

- react/src/pages/RegistroPage.tsx, función salida.
- react/src/pages/DashboardPage.tsx, registrarSalida.

Problema:

El click envía POST inmediatamente. El helper confirmAction existe, pero no se usa.
Una persona puede registrar la salida del nombre equivocado.

Corrección:

1. Pasar el objeto persona, no solo id.
2. Preguntar “¿Registrar salida de NOMBRE?”.
3. Mostrar edificio y hora de entrada.
4. Confirmar con botón carbón o rojo corporativo.
5. Cancelar sin modificar datos.
6. Mostrar éxito con nombre/hora.

Criterio:

Ninguna salida debe ocurrir con un solo click accidental.

5.2 Doble click y estado por fila
--------------------------------

Problema:

No existe exitingId/sendingId. Durante POST el botón sigue activo.

Corrección:

- const [exitingId, setExitingId] = useState<number | null>(null).
- bloquear la fila durante petición.
- texto “Registrando…”.
- finally setExitingId(null).
- no recargar toda la pantalla antes de resolver.

5.3 RoleGuard incompleto
------------------------

Problemas:

1. admin-cpf está compuesto como RoleGuard -> Shell. Si no está autorizado, el
   usuario recibe la página restringida sin sidebar/topbar ni salida visible.
2. /forbidden usa RoleGuard roles=['admin'] alrededor de “Sin permisos”; la lógica
   es confusa/invertida y no funciona como página 403 genérica.
3. Shell solo restringe sección admin. Cualquier rol autenticado desconocido ve
   Operación y Catálogos porque el else devuelve true.
4. RoleGuard usa <a href> y recarga el SPA.
5. AdminCpfPage repite isAdmin aunque la ruta ya está protegida.

Corrección recomendada:

- Cada item define roles: ['admin','registrador'] o ['admin'].
- Filtrar items por includes(user.rol).
- Mantener Shell fuera de RoleGuard:

  <ProtectedRoute>
    <Shell>
      <RoleGuard roles={['admin']}>
        <AdminCpfPage />
      </RoleGuard>
    </Shell>
  </ProtectedRoute>

- Crear ForbiddenPage simple, sin envolverla en RoleGuard.
- Usar Link para volver.
- El backend sigue siendo la autoridad; el frontend solo mejora UX.

5.4 Resultados de búsqueda vacíos
---------------------------------

Problema:

Si results es [], no se muestra “Sin resultados”. Solo desaparece la lista y queda
un hint genérico.

Corrección:

- results === null: todavía no se buscó/cargando.
- results.length > 0: lista.
- results.length === 0: alerta neutral “No encontramos coincidencias”.
- conservar el texto de búsqueda para corregirlo.

5.5 Curso/eventos falla en silencio
-----------------------------------

RegistroPage aún tiene:

.catch(() => {});

Es un campo opcional, pero si falla la API el usuario ve “Sin curso” y puede creer
que realmente no existen cursos.

Corrección:

- estado courseLoadError.
- deshabilitar select si falla.
- texto “No pudimos cargar los cursos; puede registrar sin curso”.
- permitir continuar porque es opcional.

5.6 Captions invisibles de forma incorrecta
-------------------------------------------

Las tres tablas usan:

<caption className="hidden">...</caption>

.hidden es display:none !important. Esto retira el caption del árbol accesible.

Corrección:

- crear .visually-hidden.
- cambiar caption className="visually-hidden".
- reservar .hidden para elementos realmente eliminados de layout/accesibilidad.

5.7 Semántica de tipo de persona
--------------------------------

Actualmente hay un label seguido de botones. El estado seleccionado se comunica
principalmente por color/clase.

Corrección mínima:

- fieldset + legend.
- botones con aria-pressed={tipo === value}.

Corrección ideal:

- radios visualmente personalizados dentro de labels.

5.8 Tablas no operativas en móvil
---------------------------------

.table-wrapper solo ofrece scroll horizontal. Esto evita romper el layout, pero la
acción Salida queda lejos del nombre y requiere desplazamiento lateral.

Corrección:

- desktop: tabla actual.
- móvil <=700: cards de acceso.
- cada card: nombre, tipo, edificio, entrada, estado y botón visible.
- Admin/Catálogo pueden usar cards compactas o scroll si son consulta secundaria.
- Registro/Dashboard sí requieren cards por ser operación principal.

6.1 Controles táctiles pequeños
-------------------------------

.btn--sm y .btn--icon.btn--sm miden 34 px. Para móvil/recepción táctil se recomienda
44 px.

6.2 Landing por rol
-------------------

Registrador entra al Dashboard. Su tarea principal es Registro.

Recomendación:

- admin -> Dashboard.
- registrador -> Registro.

No es un defecto de seguridad; es una mejora fuerte de facilidad.

6.3 Contexto de edificio
------------------------

El operador selecciona edificio en cada entrada. Si trabaja en un puesto fijo,
esto agrega tiempo y error.

Recomendación:

- “Edificio actual” al comenzar turno.
- preselección persistida en sessionStorage.
- visible siempre.
- modificable con confirmación.
- conservar selección tras cada registro.

Implementar solo si coincide con la operación real de la empresa.

6.4 Progressive disclosure
---------------------------

Registro muestra todos los campos de una vez. Puede guiarse mejor:

1. tipo;
2. persona;
3. edificio/detalles;
4. confirmar.

No crear un wizard lento. Usar secciones numeradas y revelar detalles después de
seleccionar persona.

6.5 Mensaje de éxito más informativo
------------------------------------

Hoy: “Entrada registrada”.

Ideal:

“Entrada registrada: Ana Pérez — Edificio Central — 08:42”.

Esto permite al recepcionista confirmar visualmente que eligió bien.

6.6 Pluralización
-----------------

Reemplazar “persona(s)” por:

- “1 persona dentro”.
- “N personas dentro”.

Si hay varias sedes, usar “en las instalaciones” o filtrar por edificio.

6.7 Lenguaje CPF/ID Referencia
------------------------------

Para usuario no técnico:

- menú “Usuarios externos”.
- título “Usuarios externos (CPF)” puede conservar sigla secundaria.
- “ID Referencia” necesita ayuda o, idealmente, un select de proveedor/instructor.

6.8 Confirmación de contraseña fuera del helper
-----------------------------------------------

AdminCpfPage importa SweetAlert directamente. Integrar el prompt de contraseña en
lib/swal.ts o componente propio para conservar tema y comportamiento.

6.9 Warning amarillo/naranja
----------------------------

SweetAlert icon:'warning' suele usar ámbar/naranja. La paleta solicitada limita los
colores a rojo/blanco/gris/negro y verde ocasional.

Opciones:

- usar icon:'question' con customClass neutral;
- usar icon:'error' rojo para acción crítica;
- personalizar completamente el icono warning a rojo/gris.

No introducir amarillo/naranja sin aprobación de marca.

6.10 Drawer: manejo de foco
---------------------------

Ya cierra con Escape y bloquea scroll, pero falta:

- mover foco al primer enlace al abrir;
- focus trap;
- devolver foco al botón menú al cerrar;
- inert/aria-hidden cuando está cerrado en móvil.

------------------

const salida = async (record) => {
  const accepted = await confirmAction(
    `Registrar salida de ${record.nombre}`,
    `${record.edificio} · Entrada ${formatTime(record.fechaEntrada)}`
  );

  if (!accepted) return;

  setExitingId(record.id);
  try {
    await api.post(`/acceso/salida/${record.id}`);
    showSuccess(`Salida registrada: ${record.nombre}`);
    await load();
  } catch {
    showError('No se pudo registrar la salida', 'Intente nuevamente.');
  } finally {
    setExitingId(null);
  }
};
